# Feature Design: Scheduling Execution Predictor

## 🎯 Objective
Empower administrators to verify their scheduling logic by providing a clear list of the next 10 execution times. This includes support for simulated predictions starting from any user-defined date and time to test complex calendar exclusions.

---

## 🏗️ Architectural Approach

### 1. API Extension (`/api/scheduling/predict`)
A new POST endpoint in the scheduling service that calculates firing times by intersecting:
- **Firing Pattern (The Schedule)**: Any supported mode (INTERVAL, CRON, MANUAL, ONE-OFF).
- **Gating Rules (The Calendar)**: All "Allow Window", "Exclude Window", and "Exception Date" rules.
- **Base Simulation Time (Optional)**: If provided, prediction starts from this timestamp. If null, it starts from `NOW()`.

### 2. Intersection Logic Flow
1. **Unfiltered Firing Sequence**: Generate the next raw sequence of potential firing times based on the **Schedule's** specific mode.
2. **Calendar Gating**: 
   - For every potential firing time, iterate through all bound **Calendars**.
   - **Gating Logic**: A run is valid ONLY if it:
     - Falls within at least one **Allow Window** (if any are defined).
     - Does NOT fall within any **Exclude Window**.
     - Does NOT fall on an **Exception Date**.
   - If blocked, the firing time is skipped and the engine evaluates the next possible occurrence from the firing pattern.
3. **Sequence Target**: Continue this process until exactly 10 valid firing times are successfully identified.

---

## 🎨 User Experience (WFA Shelf)

### A. The "Next Runs" List
Within the **Execution & Scope** section of the Workflow Admin Shelf:
- **Real-time Refresh**: Whenever the Schedule or Calendar is changed in the dropdown, the list updates automatically.
- **Visual List**:
    - `1. Mar 25, 2026 - 08:00 AM (local)`
    - `2. Mar 26, 2026 - 08:00 AM (local)`
    - ... up to 10.

### B. The "Time Machine" Mode
A "Predict from Date" input (DatePicker + TimePicker):
- **Default**: Fills with current system time.
- **Simulation**: Allows the admin to ask: *"If it's Christmas Eve at 11 PM, when will the next run happen according to my holiday exclusions?"*
- **Immediate Feedback**: The list of 10 runs updates as the user scrolls through the calendar.

## 🛡️ Edge Cases to Handle
1. **Invalid Pattern**: Clearly show a warning if the firing pattern (e.g., cron string or manual date) cannot be parsed.
2. **Search Depth Limit**: Set a "max search depth" (e.g., search up to 1000 potential runs) to avoid infinite loops if the calendar blocks all upcoming fires.
3. **Timezones**: Ensure the prediction accounts for the Workflow's timezone vs. the Calendar's specific timezone rules.
