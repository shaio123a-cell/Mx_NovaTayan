## 1) Core Concepts (Simple Mental Model)

### **Calendar**
A reusable object that defines **when it’s allowed (or forbidden) to run**.

- **Includes**: time zone, allowed time windows (e.g., Mon–Fri 08:00–18:00), exceptions (specific dates), optional exclusion/maintenance windows.
- **States**: `ACTIVE` / `PAUSED` (pausing a Calendar prevents new runs for attached schedules).
- **Impact graph**: view which workflows/schedules reference the calendar.

### **Schedule** (Timeframe)
A reusable **timeframe/pattern** that determines **when to fire**.

- **Modes** (pick one):
  1) **Once** (specific date/time)
  2) **Interval** (every N seconds/minutes/hours/days)
  3) **Weekly** (select days of week + time)
  4) **Monthly** (select days of month + time)
  5) **Cron** (advanced; optional)
- **Constraints**:
  - **Launch date** (don’t start before this date)
  - **End date** (optional)
  - **Execution limit** (`unlimited` or `max_runs = X`)
  - **Coalesce window** (avoid duplicates if multiple triggers collide; e.g., 60s)
  - **Misfire policy** on downtime: `skip`, `fire_once`, or *(Advanced)* `catch_up_all`
- **State**: `ACTIVE` / `PAUSED`.

### **Binding** (Workflow ⇄ Schedule [+ Calendar])
Attaches **Schedule + Calendar** to a **Workflow**.

- A workflow can have **multiple bindings** (keeps things flexible but simple).
- **Combination rule**: **OR** by default. Each binding can trigger the workflow; Calendar gate must allow execution.
- Optional per‑binding override: **skip if running** or **allow parallel** (`max_concurrency = N`).

> **Why this model?** Separate business windows (Calendar) from firing patterns (Schedule). Reuse independently and see impact clearly.

## 2) Behavior & Edge Cases (Clear Rules)

1) **Time zone precedence**: `Workflow > Schedule > Calendar > Tenant default`.
2) **Launch date**: compute next fire time ≥ launch date (respecting calendar).
3) **Multiple bindings** (OR): if multiple schedules fire near‑simultaneously, use **coalesce window** to emit only one run.
4) **Already running**: if binding says **skipIfRunning**, drop the trigger (audit it). Otherwise honor `max_concurrency`.
5) **Calendar gating**: even if a schedule fires, execution only if calendar is **open** at that instant. If closed, apply **misfire policy**:
   - `skip` (drop),
   - `fire_once` (emit once when next open window begins),
   - `catch_up_all` *(Advanced; hidden by default)*.
6) **Activate/Deactivate**:
   - **Calendar PAUSED** → blocks new runs for **all** bound workflows (banner on bindings).
   - **Schedule PAUSED** → blocks only that schedule’s triggers.
   - In‑flight executions continue.
7) **Execution limits**: when `max_runs` reached, auto‑pause the schedule and notify.
8) **Retries / Re‑run**:
   - **Auto‑Retry** (per binding): `max_attempts`, `retry_interval`, `strategy` = `fixed | exponential`, `jitter`.
   - **Manual Re‑run**: from Executions, **Re‑run latest** or **Re‑run from failure point** (if supported by execution graph).

## 3) Architecture (Brain = Web API)

### Components
- **Web API (App Server)** — *Scheduler brain*:
  - **Scheduler Service** (in‑process background workers)
  - **Run Queue**: DB table initially; abstract behind interface to swap to MQ later
  - **Admin REST APIs** (CRUD + preview + impact)
  - **Security**: RBAC; signed event triggers (HMAC)
- **Workers (Agents)** — *stateless executors*:
  - Poll or receive **Run Requests**
  - Execute workflow graph; post status callbacks

### Data Flow
1) Scheduler tick (~5–10s) evaluates due schedules.
2) For each due item: check **Calendar gate**, `launch/end`, `max_runs`, concurrency.
3) If allowed, enqueue **Run Request** (`PENDING`) with a **dedup_key** and **scheduled_for**.
4) Worker claims → runs → reports `SUCCESS/FAILED/CANCELED`.
5) On `FAILED` with auto‑retry enabled, Scheduler re‑enqueues at `retry_at`.

### Minimal Persistence Schema (SQL sketch)

```sql
-- Calendars
CREATE TABLE calendars (
  id              uuid PRIMARY KEY,
  name            text NOT NULL,
  timezone        text NOT NULL,
  state           text NOT NULL CHECK (state IN ('ACTIVE','PAUSED')),
  created_by      text,
  updated_at      timestamptz DEFAULT now()
);

CREATE TABLE calendar_rules (
  id           uuid PRIMARY KEY,
  calendar_id  uuid REFERENCES calendars(id) ON DELETE CASCADE,
  rule_type    text NOT NULL, -- ALLOW_WINDOW | EXCLUDE_WINDOW | EXCEPTION_DATE | HOLIDAY_SET_REF
  payload_json jsonb NOT NULL
);

-- Schedules (Timeframes)
CREATE TABLE schedules (
  id              uuid PRIMARY KEY,
  name            text NOT NULL,
  mode            text NOT NULL, -- ONCE | INTERVAL | WEEKLY | MONTHLY | CRON
  payload_json    jsonb NOT NULL,
  launch_at       timestamptz,
  end_at          timestamptz,
  max_runs        int,
  misfire_policy  text NOT NULL DEFAULT 'fire_once', -- skip | fire_once | catch_up_all
  coalesce_secs   int DEFAULT 60,
  state           text NOT NULL CHECK (state IN ('ACTIVE','PAUSED')),
  updated_at      timestamptz DEFAULT now()
);

-- Binding Workflow <-> Schedule (+ Calendar)
CREATE TABLE workflow_schedule_bindings (
  id               uuid PRIMARY KEY,
  workflow_id      uuid NOT NULL,
  schedule_id      uuid REFERENCES schedules(id) ON DELETE CASCADE,
  calendar_id      uuid REFERENCES calendars(id),
  skip_if_running  boolean DEFAULT true,
  max_concurrency  int DEFAULT 1,
  retry_policy_json jsonb,
  state            text NOT NULL CHECK (state IN ('ACTIVE','PAUSED')),
  updated_at       timestamptz DEFAULT now()
);

-- Run queue
CREATE TABLE workflow_runs (
  id             uuid PRIMARY KEY,
  workflow_id    uuid NOT NULL,
  binding_id     uuid REFERENCES workflow_schedule_bindings(id),
  scheduled_for  timestamptz NOT NULL,
  trigger_source text NOT NULL, -- schedule|event|manual
  dedup_key      text,
  status         text NOT NULL CHECK (status IN ('PENDING','RUNNING','SUCCESS','FAILED','CANCELED')),
  attempt        int DEFAULT 0,
  payload_json   jsonb,
  created_at     timestamptz DEFAULT now(),
  started_at     timestamptz,
  finished_at    timestamptz,
  error          text
);
```

## 4) REST API (CRUD‑first)

### Calendars
```http
POST   /api/calendars
GET    /api/calendars
GET    /api/calendars/{id}
PUT    /api/calendars/{id}
PATCH  /api/calendars/{id}/state   { "state": "ACTIVE|PAUSED" }
GET    /api/calendars/{id}/usage   -- workflows & schedules referencing it
POST   /api/calendars/{id}/preview -- body: time range; returns open/closed intervals
```

**Calendar payload example**
```json
{
  "name": "IL Business Hours",
  "timezone": "Asia/Jerusalem",
  "state": "ACTIVE",
  "rules": [
    {"type":"ALLOW_WINDOW","daysOfWeek":[0,1,2,3,4],"start":"08:00","end":"18:00"},
    {"type":"EXCEPTION_DATE","date":"2026-04-23","reason":"Holiday"}
  ]
}
```

### Schedules (Timeframes)
```http
POST   /api/schedules
GET    /api/schedules
GET    /api/schedules/{id}
PUT    /api/schedules/{id}
PATCH  /api/schedules/{id}/state    { "state": "ACTIVE|PAUSED" }
POST   /api/schedules/{id}/preview  -- returns next N fire times given tz + launch/end
```

**Schedule payload examples**
```json
{
  "name": "Every 15 Minutes from Launch",
  "mode": "INTERVAL",
  "payload": {"every":"15m"},
  "launchAt": "2026-03-20T08:00:00+02:00",
  "endAt": null,
  "maxRuns": null,
  "misfirePolicy": "fire_once",

```


```json

```

### Bindings (attach Schedule+Calendar to Workflow)
```http

```

## 5) UX Design (Keep it n8n‑simple)

### Side Menu: **Calendars & Schedules** (new)
- **List**: Name, Type (Calendar/Schedule), State, Last Edited, **Usage count**.
- **Calendar Detail**:
  - Tabs: *Rules*, *Preview*, *Usage*, *Audit*.
  - **Rules**: Add ALLOW/EXCLUDE windows, exception dates; timezone picker.
  - **Preview**: show open/closed slots for next 2 weeks.
  - **Usage**: workflows/bindings using this calendar; drill‑down into Workflow Admin.
- **Schedule Detail**:
  - Mode‑driven minimal form (Once/Interval/Weekly/Monthly; Cron under *Advanced*).
  - **Advanced** accordion: misfire policy, coalesce, execution limit.
  - **Preview**: next N fire times (with calendar applied & launch date respected).

### Workflow Admin → **Scheduling** Tab (new)
- **Bindings list** (cards):
  - Schedule summary (e.g., “Every 15m from 08:00”), Calendar chip (“IL Business Hours”), **State** toggle.
  - Retry policy summary; actions: Pause/Resume, Edit, Remove, **Preview next 10**.
- **Add Binding** wizard:
  1) Pick **Schedule** (dropdown)
  2) Pick **Calendar** (optional; defaults to “Always Open”)
  3) Options: skipIfRunning, maxConcurrency, retry policy (collapsed by default)
- **Execution Controls**: Manual Run (now), Manual Re‑run last failed, **Disable scheduling** master switch.

## 6) Retry / Re‑run Strategy
- **Auto‑Retry** (default): `maxAttempts=3`, `interval=5m`, `strategy=fixed`.
- **Manual Re‑run**: re‑use last input context or re‑run from failed step (if checkpoints supported).
- **Escalation**: after final failure, notify (email/Slack/Teams) and mark `FAILED_FINAL`.

---

## 7) Observability & Impact Analysis
- **Calendar Usage** page: count + list of dependent workflows/bindings; audit trail.
- **Change impact banner** when editing a calendar used elsewhere.
- **Metrics**: runs triggered, skipped (calendar closed), skipped (already running), retries, misfires.

---

## 8) Security & Governance (Lightweight)
- **RBAC**: `CalendarAdmin`, `ScheduleAdmin`, `WorkflowAdmin` roles.
- **API triggers**: HMAC‑signed; per‑workflow secrets; optional IP allowlist.
- **Audit**: CRUD on calendars/schedules/bindings, state toggles.

---

## 9) Why one calendar per binding & multiple bindings per workflow?
- Provides **more than n8n** flexibility without complexity.
- **OR semantics** remain intuitive.
- Fine control via **skipIfRunning**, `maxConcurrency`, and **coalescing**.
- If needed later, support `combine = "OR" | "AND" | "EXCEPT"` (Advanced) — **defer for now**.

---

## 10) MVP / Phased Delivery

**Phase 1 (MVP)**
- Calendars (ALLOW + exceptions, ACTIVE/PAUSED)
- Schedules: Once, Interval, Weekly, Monthly; Launch date; Execution limit; Coalesce; Misfire = `skip|fire_once`
- Bindings (one calendar per binding; multiple per workflow; OR logic)
- Retry policy (fixed backoff), skipIfRunning
- Previews + Usage lists
- Event API trigger (signed)

**Phase 2**
- Cron mode; Exclude windows; Holiday sets
- Concurrency >1; per‑workflow `max_concurrent_runs`
- Manual “Re‑run from failed step”
- Impact preview banner + audit UI

**Phase 3**
- Misfire `catch_up_all` (Advanced)
- Blackout sets; iCal import
- SLA alerts; richer notifications
