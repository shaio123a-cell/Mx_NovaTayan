# Feature Design: REST Webhook Triggers

## đźŽŻ Objective
Enable real-time, event-driven workflow execution by exposing secure, unique HTTP endpoints (Webhooks). This allows external systems (SaaS apps, monitoring tools, custom APIs) to trigger business processes by sending a JSON payload.

## đźŹ—ď¸Ź Architectural Approach

### 1. The "Trigger Registration" Model
A new **WebhookTrigger** entity in the database that ties a unique, cryptographically secure token to a specific Workflow.
- **`id`**: Unique identifier.
- **`tokenId`**: A secret string (e.g., `whk_abc123...`) used in the URL.
- **`workflowId`**: The workflow to execute.
- **`state`**: ACTIVE / INACTIVE.
- **`mapping`**: A JSON object defining how properties from the incoming HTTP request (headers, query params, body) map to the workflow's **Input Variables**.

### 2. The Inbound Gateway (`POST /triggers/:token`)
A public, un-authenticated endpoint that:
1.  **Validates the Token**: Finds the associated `WebhookTrigger`.
2.  **Context Construction**: 
    - Merges the incoming `body`, `query`, and `headers` into a single "Source Data" object.
3.  **Variable Transformation**: 
    - Applies mapping logic (e.g., `SourceData.body.customer.id` â†’ `Workflow.Input.clientId`).
4.  **Execution Enqueuing**: Starts the workflow execution with the mapped variables.
5.  **Response**: Returns the `executionId` and a `202 Accepted` status.

## đźŽ¨ User Experience (WFA Shelf)

### A. The "Webhooks" Tab
A new tab next to **Scheduling** in the Workflow Admin shelf where users can:
- **Enable/Disable** Webhook triggers for the specific workflow.
- **Copy Webhook URL**: A pre-formatted URL like `http://api.myapp.com/triggers/whk_4f92...`.
- **Sample Request (cURL)**: A code block showing exactly how to call the trigger from terminal/Postman.

### B. Payload Mapping UI
A simplified mapping editor where the user can:
- Define which JSON key from the incoming request maps to which **Workflow Input Variable**.
- Choose where the data comes from (`BODY`, `QUERY`, or `HEADER`).

## đź›ˇď¸Ź Edge Cases & Security
1.  **Token Rotation**: Allow users to regenerate a token if it's accidentally leaked publicly.
2.  **Rate Limiting**: Throttling trigger frequency per token to prevent DDoS or accidental loops.
3.  **Idempotency**: Accept an `X-Idempotency-Key` header so that duplicate HTTP retries from the caller don't trigger the same workflow twice.
